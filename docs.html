Documentation

User guide for the Detection Engineering Console — a centralized dashboard for managing SIEM detection rules across multiple customers and platforms.

Overview

The console is organized into three main areas accessible from the sidebar:

Dashboard

The home page. Displays a real-time summary of all detection rules — rule counts by status and severity, and a table of the 5 most recently modified rules.

Detection Rules

The main working area. Create, edit, delete, search, and filter all detection rules. Each rule is linked to a customer and SIEM platform via Settings.

Settings

Configure the reference data that rules depend on:

  • SIEM Platforms — Define which SIEM solutions are used (e.g., Elastic, Splunk, Microsoft Sentinel)
  • Customers — Create customer profiles and assign each to a SIEM platform
Set up SIEM Platforms and Customers before creating detection rules, as rules require both to be configured first.

Dashboard

The dashboard gives an at-a-glance view of your rule inventory without requiring any interaction.

Rules by Status

Five stat cards showing rule counts: Total, Production, Testing, Draft, and Deprecated.

Rules by Severity

Four stat cards showing counts by severity level: Critical, High, Medium, and Low.

Recently Modified

A table showing the 5 most recently modified rules, sorted by Modified Date descending. Columns: Rule Name, Customer, Platform, Severity, Status, Modified, Modified By.

Detection Rules

Adding a Rule

1 Go to Detection Rules — Click "Detection Rules" in the sidebar
2 Click "Add Rule" — Button at the top right
3 Fill in the form — Fields marked with * are required. See Rule Fields for details.
4 Click "Save Rule" — Rule is saved and you are returned to the Detection Rules list

Editing a Rule

Click the pencil icon on any row in the rules table. The form opens pre-filled with the current values. Make changes and click "Update Rule".

Deleting a Rule

Click the trash icon on any row. A confirmation dialog will appear before deletion. This action cannot be undone.

Rules Table Columns

The table displays: No, Rule Name, Customer, Platform, Tactic, Technique, Severity, Status, Created, Modified, Created By, Modified By, and Actions. Dates are shown in human-readable format (e.g., "17 Apr 2026"). The table scrolls horizontally on smaller screens.

Rule Fields Reference

Complete reference for every field in the Add / Edit Rule form.

Field Required Description
Customer Required The customer this rule is deployed for. Must be pre-configured in Settings → Customers.
Rule Name Required A clear, descriptive name for the detection. Max 200 characters.
Detection Logic Required The actual query or logic used in the SIEM to detect the threat (e.g., SPL for Splunk, KQL for Elastic/Sentinel).
Created By Required Name or identifier of the person who created this rule. Used for audit trail.
Modified By Required Name or identifier of the person who last modified this rule. Update this field whenever you edit the rule.
MITRE Tactic Optional ATT&CK tactic this rule maps to. Options: Reconnaissance, Resource Development, Initial Access, Execution, Persistence, Privilege Escalation, Defense Evasion, Credential Access, Discovery, Lateral Movement, Collection, Command and Control, Exfiltration, Impact.
MITRE Technique Optional Technique ID from ATT&CK, e.g., T1078. Max 100 characters.
Severity Default: Low Risk level of the detection. Options: Critical High Medium Low
Status Default: Draft Lifecycle stage of the rule. Options: Draft Testing Production Deprecated
Created Date Default: Today Date the rule was originally created. Defaults to today's date.
Modified Date Default: Today Date of the last modification. Update this when editing the rule.

Filtering & Searching Rules

The filter panel on the Detection Rules page lets you narrow down the rules table using multiple criteria simultaneously. Active filters are shown as tags below the filter panel.

Search Bar

Full-text search across Rule Name and MITRE Technique fields. Supports partial matches.

Dropdown Filters

  • Customer — Filter rules belonging to a specific customer
  • SIEM — Filter rules by the customer's SIEM platform
  • Tactic — Filter by MITRE ATT&CK tactic (all 14 tactics available)
  • Severity — Filter by severity: Critical, High, Medium, or Low
  • Status — Filter by lifecycle status: Draft, Testing, Production, or Deprecated

Date Range Filters

  • Created From → To — Filter rules by Created Date range
  • Modified From → To — Filter rules by Modified Date range

You can set only a start date, only an end date, or both to define a range.

Text Filters

  • Created By — Partial match search on the Created By field
  • Modified By — Partial match search on the Modified By field
  • MITRE Technique — Partial match search on the Technique field

Applying and Clearing Filters

  • Click "Filter" to apply the selected criteria
  • Click "Clear All" (appears when filters are active) to reset all filters
  • The result counter "Showing X of Y rules" updates to reflect active filters

Settings

Settings contains two tabs: SIEM Platforms and Customers. Configure these before creating any rules.

SIEM Platforms

Manage the list of SIEM solutions used in your environment.

  • Add — Click "Add Platform", enter a name (e.g., Elastic, Splunk, Microsoft Sentinel), and click Save
  • Edit — Click the pencil icon on a platform row to rename it
  • Delete — Click the trash icon. Deletion is blocked if any Customer is still assigned to this platform
Note: Deleting a SIEM platform that has customers assigned will show an error message. Remove or reassign all customers from that platform first.

Customers

Manage customer profiles. Each customer is assigned to exactly one SIEM platform.

  • Add — Click "Add Customer", enter a name and select a SIEM platform, then click Save
  • Edit — Click the pencil icon to update a customer's name or SIEM assignment
  • Delete — Click the trash icon. Deletion is blocked if any Detection Rule references this customer
Note: Deleting a customer that has rules assigned will show an error message. Delete or reassign all rules for that customer first.

Rule Status Workflow

Each rule moves through a lifecycle. Status is updated manually via the Edit Rule form.

Draft → Testing → Production → Deprecated

Status Descriptions

  • Draft — Rule is being written or refined. Not yet deployed to SIEM. Use this when first creating a rule.
  • Testing — Rule is deployed in a test or staging SIEM environment. Validate for false positives and logic correctness before promoting.
  • Production — Rule is live and actively detecting threats in production SIEM. Detection logic must be complete and verified.
  • Deprecated — Rule has been retired. Kept for historical reference. No longer active in SIEM.
Status does not automatically change — you must edit the rule and update it manually. This gives full control over the promotion process.

Best Practices

Naming Rules

  • Use a consistent naming pattern, e.g.: [Tactic] - [Threat] - [Platform]
  • Be specific — "Brute Force Login via SSH" is better than "Login Detection"
  • Avoid abbreviations that are not widely understood

MITRE ATT&CK Mapping

  • Always fill in Tactic and Technique when the threat maps to ATT&CK — it enables structured filtering
  • Use the official technique ID format: T1078 for techniques, T1078.001 for sub-techniques

Detection Logic

  • Write the query in the syntax of the target SIEM (SPL for Splunk, KQL for Elastic/Sentinel, etc.)
  • Include comments in the logic to explain non-obvious conditions
  • Test the query in the SIEM before promoting to Production status

Audit Trail

  • Always fill in Created By when adding a rule
  • Update Modified By and Modified Date every time you edit a rule — this keeps the audit trail accurate

Status Progression

  1. Create the rule in Draft — write and refine the detection logic
  2. Promote to Testing — deploy to staging SIEM and monitor for false positives
  3. Promote to Production — only after validation is complete
  4. Move to Deprecated — when a rule is replaced or no longer relevant (do not delete, keep for history)
